## Mission

Three gaps in v0.1.5's user experience:

1. **No `ael update`** â€” users who installed v0.1.5 have no way to
   upgrade except re-running the curl one-liner from scratch. Most
   users don't even know that's the path.
2. **No `ael uninstall`** â€” README FAQ tells users to run
   `aiplus uninstall --yes` (substrate leak). Users have no clean way
   to roll back.
3. **No first-run onboarding** â€” after `ael install` completes, the
   terminal is black. New users don't know what to do next, where
   their team is, or whether install actually worked.

Track B closes all three.

## Scope (4 bundles, B1 / B2 / B3 / B4)

### B1. `ael update` command

Acceptance criteria:

- B1.1. New top-level subcommand `ael update`.
- B1.2. Behavior: fetch `/releases/latest` â†’ compare to installed
  version (read from `ael --version`) â†’ if same, print "already
  up-to-date" and exit 0 â†’ if different, download new tarball,
  verify sha256, replace `~/.local/bin/ael` and `~/.local/libexec/ael-support`
  atomically (write to .new then mv).
- B1.3. `--dry-run` flag shows what would happen without writing.
- B1.4. Respects same env vars as install.sh: `AEL_INSTALL_DIR`,
  `AEL_LIBEXEC_DIR`.
- B1.5. Print version diff: "AEL 0.1.5 â†’ AEL 0.1.6".
- B1.6. Test: `ael update --dry-run` in a fake project shows correct
  upgrade path.

### B2. `ael uninstall` command

Acceptance criteria:

- B2.1. New top-level subcommand `ael uninstall`.
- B2.2. Default behavior: remove ONLY the user-installed `ael` binary
  and `ael-support` helper. Do NOT touch project-level `.aiplus/`
  directories (preserves user persona customizations).
- B2.3. `--purge` flag: also remove the project-level `.aiplus/` in
  the current directory (one project at a time, no global cleanup).
- B2.4. Interactive confirmation prompt unless `--yes`.
- B2.5. Print clear summary of what was removed.
- B2.6. README FAQ updated to say `ael uninstall --yes` (the existing
  substrate-leak reference to `aiplus uninstall --yes` is replaced â€”
  coordinate with Track C, who's editing README anyway).

### B3. First-run onboarding

After `ael install` completes successfully in a new project, the user
needs a sane "what now" experience.

Acceptance criteria:

- B3.1. `ael install` final output, in addition to "Next: ael",
  prints a 3-step onboarding hint:
  ```
  Quick start (your team is ready):

    ael                  open the lobby â€” pick who to talk to (PI / Advisor / â€¦)
    ael advisor          jump straight to Advisor for paper framing
    ael pi               jump straight to PI for orchestration

  Tip: open two terminal windows for the Ownerâ†’Advisor + Ownerâ†’PI flow
  (one window per role).

  More: ael --help
  ```
- B3.2. New env var `AEL_NO_ONBOARDING=1` to suppress in non-interactive
  contexts (CI, scripts).
- B3.3. First-time-only welcome: on first `ael` invocation in a
  project (no `.aiplus/.ael-greeted` marker), print a one-line
  welcome before entering the lobby. Subsequent invocations skip it.

### B4. install.sh PATH auto-write (opt-in)

If `~/.local/bin` is not in `$PATH`, `install.sh` currently prints a
manual fix instruction. Add an opt-in flag that writes it for the user.

Acceptance criteria:

- B4.1. New `install.sh` flag `--add-to-path`.
- B4.2. When set AND `~/.local/bin` is not in `$PATH`, append a single
  `export PATH=...` line to the appropriate shell profile:
  - zsh â†’ `~/.zshrc`
  - bash â†’ `~/.bashrc`
  - fish â†’ `~/.config/fish/config.fish` with appropriate syntax
- B4.3. The append is idempotent: re-running `--add-to-path` does not
  add a second line. Detect existing `export PATH=` with our exact
  string and skip.
- B4.4. Print clear "appended to ~/.zshrc" message + "restart your
  shell or run: source ~/.zshrc" hint.
- B4.5. Without `--add-to-path`, behavior is identical to today
  (print manual instruction).
- B4.6. Test in CI: install with `--add-to-path` to a tmpdir HOME, verify
  `.zshrc` has exactly one line added.

## Halt conditions

- S1. B1 download-and-replace requires the running `ael` process not
  to be the same file being replaced. If user runs `ael update` from
  the wrapper itself, we can't overwrite the open binary on Linux/macOS.
  Stop and ask: write to `.new` and rename + exec into the new copy,
  or instruct user to run from a different terminal?
- S2. B3 first-time welcome marker file `.aiplus/.ael-greeted` â€”
  conflict with substrate hidden-dot convention? Stop and ask: use
  `.ael/.greeted` instead, or accept `.aiplus/` for now until v0.3
  rename.
- S3. B4 shell profile detection: user's `$SHELL` may not match what
  they actually use. Stop and ask: which heuristic to prefer ($SHELL,
  $0, ps -p $PPID)?
