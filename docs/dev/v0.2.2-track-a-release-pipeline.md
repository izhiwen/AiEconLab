# AEL v0.2.2 · Track A — Release Pipeline & CI Coverage

CEO autonomous-loop goal file for **Track A** of the v0.2.2 sprint.
Run in parallel with Track B and Track C (they don't touch the same
files). See `v0.2.2-tracks-overview.md` for the bigger picture.

## Mission

The AEL release pipeline currently:

1. Hardcodes the version number in 5 separate files (`install.sh`,
   `ael` wrapper, `scripts/build-ael.sh`, two test files). Every bump
   requires editing all five; we've already shipped a wrong-version
   release once (v0.1.4 tarball had a 0.1.3 banner) because of this.
2. Only builds **2 of 4 platforms** users actually have: darwin-aarch64
   and linux-x86_64. Intel Mac and ARM Linux users get a hard "no
   package available" failure from `install.sh`.
3. Has no automated install-smoke for macOS — we ship darwin-aarch64
   tarballs blind. The fact that v0.1.5 install works on Apple Silicon
   is currently faith-based, not verified.

Track A closes all three.

## Scope (3 bundles, A1 / A2 / A3)

### A1. install.sh auto-detect latest release

Replace the hardcoded `AEL_DEFAULT_VERSION="v0.1.5"` constant with a
runtime lookup against GitHub's `/releases/latest` redirect endpoint.

Acceptance criteria:

- A1.1. `install.sh` does NOT contain a hardcoded version string. The
  default version comes from:
  ```sh
  AEL_DEFAULT_VERSION="$(
    curl -fsSL -o /dev/null -w '%{url_effective}' \
      "https://github.com/$REPO/releases/latest" \
      | sed -E 's#^.*/tag/##'
  )"
  ```
  Or equivalent — point is no v0.1.x literal in the source.
- A1.2. The lookup is cached for the duration of the install (single
  curl, not one per asset).
- A1.3. If the curl fails (e.g. user offline, GitHub rate-limited),
  fall back to a "MINIMUM_SUPPORTED" hardcoded version with a clear
  warning. The fallback must be conservative (older than current
  latest) — never "newer than what exists".
- A1.4. `--dry-run` still prints `version=vX.Y.Z`. New users running
  `curl ... | bash` see the resolved latest version, not "${LATEST}"
  literal.

### A2. 4-platform release matrix

Currently `release-ael.yml` builds:

- ubuntu-22.04 → linux-x86_64
- macos-latest (macos-14) → darwin-aarch64

Add:

- macos-14 with `--target x86_64-apple-darwin` → darwin-x86_64
- ubuntu-22.04 with `cross` → linux-aarch64

Acceptance criteria:

- A2.1. `release-ael.yml` matrix has 4 entries producing 4 tarballs
  + 4 sha256 sidecars = 8 release assets per cut.
- A2.2. `scripts/build-ael.sh` accepts an optional `TARGET=<rust-triple>`
  env var. When set, `cargo build --target $TARGET` is used and the
  package asset name uses POSIX naming (e.g. `darwin-x86_64`,
  `linux-aarch64`) regardless of the rust triple format.
- A2.3. `install.sh`'s `detect_asset` already handles the 4 POSIX
  combos (already does); verify it doesn't need changes.
- A2.4. Cross-rs configuration: add `Cross.toml` if not present;
  configure the ubuntu-aarch64 image with libdbus-1-dev install.
- A2.5. Cache cargo per (os, target) so the matrix doesn't double the
  CI time linearly.

### A3. macOS install-smoke CI

`install-smoke.yml` currently runs on ubuntu-latest across 3 runtimes.
There is **no automated test that `ael install codex` actually works
on macOS** — we discover macOS bugs only when users try it.

Acceptance criteria:

- A3.1. New job `install-smoke-macos` in `install-smoke.yml` running
  on macos-14 (Apple Silicon).
- A3.2. The job builds aiplus from source (same pattern as the linux
  smoke), then runs the full `aiplus install codex + add aieconlab +
  doctor` flow.
- A3.3. macOS smoke only on `codex` runtime (claude-code and opencode
  on macOS would need their own CLIs which are not available in
  hosted runners; skip).
- A3.4. Cache cargo to keep cold start under 12 min.

## Halt conditions (CEO stops + asks Owner)

- S1. A1 `/releases/latest` redirect lookup turns out to be rate-limited
  for unauthenticated curl. Stop and ask: should we switch to GitHub
  API with retries, or pin a static fallback version?
- S2. A2 `cross` Docker build for aarch64-unknown-linux-gnu fails for
  a non-trivial reason (e.g. libdbus pre-build hook collides). Stop
  and ask: should we drop linux-aarch64 from the matrix for now, or
  invest in fixing?
- S3. A3 macOS smoke takes longer than 20 min even with cache (because
  cargo target caching has 1st-build cold start). Stop and ask: trim
  scope or accept slower CI?

## Anti-patterns

- Do NOT bump the AEL version while doing this. Track A lands changes
  to main only; release cutting is Owner's decision.
- Do NOT modify `ael` wrapper logic. That's Track B's territory.
- Do NOT touch README. That's Track C's territory.
- Do NOT consolidate the 3 PRs into one — keep A1, A2, A3 as separate
  PRs for clean review and atomic rollback.
