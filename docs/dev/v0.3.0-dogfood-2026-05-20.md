## Findings

### D1. Lobby setup line leaks substrate brand

Transcript:

```text
✦ aiplus: project is set up.
```

Expected: `AEL` branding, consistent with the next line:

```text
✦ AEL Lobby — A1_Qing_Ideology
```

Likely owner: aiplus substrate. The line is printed before lobby render and
does not use `AIPLUS_BRAND`.

Severity: cosmetic but visible; v1.0 brand acceptance failure.

### D2. `ael pi --fresh` fails in installed dogfood because PATH is stale

Transcript:

```text
ael pi --fresh
error: unexpected argument '--fresh' found
```

Expected: open a clean PI session. `--fresh` is an AEL wrapper flag and should
not leak to `aiplus agent talk`.

Diagnosis: `/Users/steve/.local/bin/ael` is still `AEL 0.2.10`. The installed
0.2.10 wrapper delegates role shortcuts as `agent talk "$role" "$@"`, so
`--fresh` leaks into aiplus. The 0.3.0 source wrapper strips `--fresh`.

Severity: dogfood setup/PATH drift, not a confirmed 0.3.0 source regression.

### D3. Direct natural-language routing does not reach `role_aliases` in stale wrapper

Transcript:

```text
ael "课题负责人"
error: unrecognized subcommand '课题负责人'

ael "我想反思 RD 设计"
error: unrecognized subcommand '我想反思 RD 设计'
```

Expected: AEL should delegate a single freeform argument to
`aiplus agent talk <input>` so aiplus v0.6.20 can resolve `[role_aliases]`.

Do not restore or expand bash `detect_role_from_input` for this path; the
dogfood evidence says the wrapper is bypassing aiplus role alias resolution,
not that role aliases failed.

Diagnosis: the installed 0.2.10 wrapper passes unknown top-level args through
to `aiplus` directly. The dev-branch hotfix now delegates one freeform argument
to `aiplus agent talk <input>`.

Severity: dogfood setup/PATH drift plus wrapper gap; fixed in source by this
hotfix.

### D4. `agent_route` records dispatch but does not create an executable artifact

PI correctly called `agent_route` for `ra-python`. `agent_status` then showed
`ra-python` active, but worktree remained `[missing]` and no requested report
file appeared after two dispatches.

Observed:

- Dispatch log entries exist.
- Active roles include `ra-python`.
- Worktree status remains `0 of 20 provisioned`.
- Requested artifact
  `AIER_OS_Qing/results/manuscript-staging/reports/docs_inventory_2026-05-20.md`
  does not exist.

This is worse than a slow agent: the UI implies work was sent, but no worker
actually ran. The PI's recovery behavior was good: it checked status and the
target artifact instead of claiming success.

Likely owner: aiplus agent-team substrate / MCP dispatch semantics.

Severity: high for coordinator trust.

### D5. Cross-role transcript memory gap reproduced in real dogfood

Advisor gave a substantive identification critique. A later PI session could
not recover it and reported no Advisor memory/team memory/dispatch record.

This is a real incident of cross-role continuity gap. It should count toward
the cross-role memory bridge dogfood evidence, but it is not by itself enough
to unpark cross-runtime handoff. The failed recall is not about literal runtime
resume; it is about role output not being summarized into memory.

Severity: expected v1.1 gap, now confirmed in a real project.

### D6. Runtime selection is inconsistent across entry paths

Lobby path allowed explicit runtime selection and launched PI under Codex.
Direct `ael advisor` and `ael pi` later launched Claude Code by default.

This may be acceptable if manifest/runtime preference says Claude, but it is
friction in dogfood because the Owner's mental model was "using Codex". Track
whether this causes repeated confusion before changing behavior.

Severity: UX friction; not yet a release blocker.

## Installed Wrapper Drift Runbook

The dogfood terminal used `/Users/steve/.local/bin/ael`, which reported
`AEL 0.2.10`, while this repo source reports `AEL 0.3.0 (aiplus 0.6.19+)`.
Before the next dogfood pass, verify the installed binary is the intended one:

```sh
command -v ael
ael --version
./ael --version
```

If `ael --version` is still older than `./ael --version`, use the release/update
path rather than judging dogfood behavior from the stale wrapper:

```sh
ael update --dry-run
```

If the desired 0.3.0 package has not been published yet, treat the session as a
source-wrapper dogfood by invoking `./ael` from this repo or by publishing the
0.3.0 package first. Do not classify stale-installed-wrapper behavior as a
fresh 0.3.0 regression.
