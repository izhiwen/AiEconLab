# v0.3.0 Memory Audit

Date: 2026-05-20
Scratch project: `/tmp/ael-memory-audit-u4u6`
Runtime: Codex CLI 0.131.0
Substrate: vendored aiplus v0.6.19

## Setup

The audit used a clean temporary project with AEL installed through the
vendored v0.6.19 binary.
Initial `aiplus memory status` reported project-local memory installed,
zero active records, no cloud sync, no automatic transcript learning, no
secret values, and no global agent config edits.
The install seeded `.aiplus/agent-memory/*`, but role directories only
contained README and placeholder files at startup.

## M1: Writer Continuity

Scenario: open `ael writer`, draft three RD identification paragraphs with
marker `MEMORY_AUDIT_WRITER_RD`, then open a later writer session and ask
what was written.
The first writer loaded persona context with `memory_loaded=yes`.
The writer manually stored the marker using `aiplus memory add --scope
project`, creating one active record in `.aiplus/memory/project-memory.jsonl`.
The later writer loaded project memory and quoted the marker exactly.
Confound: the audit draft also existed in a scratch project file, so full
draft recall came from filesystem search as well as memory.
Conclusion: explicit project memory survives across writer sessions, but
ordinary transcript recall is not automatic and project files can mask that.

## M2: Cross-Role Isolation

Scenario: advisor critiques a weak-IV story with marker
`MEMORY_AUDIT_ADVISOR_IV`; the capture is stored outside the project; a
fresh writer is asked what the advisor flagged.
The advisor saw the existing M1 project-memory RD record.
The advisor critique was not written to shared memory or per-role memory.
The later writer did not recover the advisor IV marker or critique.
It only verified the shared writer/RD project-memory record.
Conclusion: cross-role bleed is controlled when outputs are not stored in
shared memory or project files. Advisor-to-writer handoff is not automatic.

## M3: PI Dispatch Recall

Scenario: PI dispatches to `ra-stata` with marker
`MEMORY_AUDIT_PI_DISPATCH`; after more than five minutes, a fresh PI session
is asked for the last dispatched task.
The PI routed a first-stage regression audit to `ra-stata`.
AiPlus wrote `.aiplus/agents/dispatch-log.jsonl` with role `ra-stata`, tier
`MEDIUM`, timestamp `2026-05-20T17:05:36.956Z`, and the exact marker.
The fresh PI used `agent_status`, `agent_audit_verify_log`, and the dispatch
log to recover the role, task text, and marker exactly.
Conclusion: coordinator recall works through structured dispatch metadata,
not opaque transcript memory. Audit verification passed for the dispatch log.

## What Works

Project-local memory is installed by default and visible to role sessions.
Explicit memory writes survive across fresh sessions.
Dispatch metadata is durable enough for PI status recall.
Dispatch records include role, tier, timestamp, task text, and hash data.
Shared memory does not ingest every role response, limiting accidental bleed.

## Gaps

Automatic transcript learning is still reported as `none`.
Per-role memory directories are seeded but not populated by ordinary chat.
Advisor output is invisible to writer unless explicitly shared.
Memory CLI help is loose; agents made wrong first attempts before valid calls.
M1 is a partial recall proof because a project file contained the draft.

## Quick Wins

Document `aiplus memory add` for role sessions; some guidance still says
`record`.
Mirror compact PI dispatch summaries into team memory.
Add a first-class "last dispatch" command with text and JSON output.
Expose an `ael memory status` wrapper path for user verification.
Keep future audit captures outside the project to separate memory from search.
