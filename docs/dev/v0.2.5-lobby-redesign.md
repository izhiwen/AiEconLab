# AEL v0.2.5 · Lobby Redesign — Auto-Install + Runtime Picker

CEO autonomous-loop goal file. Companion to the short `/goal` text.

## Mission

Today (2026-05-19) Owner discovered the current `ael` lobby UX has
two friction points:

1. **Project isn't set up → ael errors** asking user to run `ael
   install`. Friction: user has to remember a second command before
   they can even start talking.
2. **Lobby only asks role, not runtime**. The auto-detected runtime
   may not be the one the user wants. Owner specifically wanted to
   reach Advisor via Claude Code but ael picked codex by default.

v0.2.5 closes both. After this sprint, the **entire onboarding** from
"I have AEL installed" to "I'm in PI chat" is **one word — `ael` —
plus two short answers**:

```
$ cd MyPaper
$ ael
  → ael auto-installs adapters for whichever runtimes are on PATH
  → "Who do you want to talk to?"
  > pi
  → "Which runtime? codex / claude-code / opencode"   [skipped if 1 only]
  > claude
  → [Claude Code opens with PI persona loaded]
```

## Owner-confirmed design decisions

| Question | Owner choice |
|---|---|
| Q1. Always ask runtime, or only when multiple available? | **Hybrid**: 1 installed → skip; ≥2 installed → ask |
| Q2. If a runtime CLI isn't on PATH? | **Silently skip** that runtime during auto-install (no error, no warning) |
| Q3. Where do these changes ship? | **v0.2.5 sprint** (this file), not bundled with v0.2.4 |

## Scope (3 bundles)

### L1. Auto-install on first `ael` invocation

Trigger condition: `ael` (no args) OR `ael <role>` shortcut OR `ael
chat`, executed in a directory that does NOT yet have
`.aiplus/manifest.json`.

Acceptance criteria:

- L1.1. Replace the current `chat_default` "exit with hint" early
  return with an auto-install path:
  - For each of [`codex`, `claude-code`, `opencode`]:
    - If `command -v <runtime-cli>` succeeds (`codex` / `claude` /
      `opencode` binaries on PATH), run the equivalent of
      `ael install <runtime>` (install adapter + add aieconlab +
      set-team aieconlab + mcp-register).
    - If the CLI is NOT on PATH, **silently skip** (no error, no
      warning, no per-runtime mention).
- L1.2. After auto-install, print one summary line:
  ```
  AEL set up for: codex, claude-code, opencode    # or whichever subset
  ```
- L1.3. If ZERO runtime CLIs found on PATH, print a clear error and
  exit non-zero — user has to install at least one of Claude Code /
  Codex / OpenCode before AEL is usable. Provide download URLs.
- L1.4. Subsequent `ael` invocations (manifest exists) skip the
  auto-install path entirely. No latency penalty for established
  projects.
- L1.5. If `.aiplus/manifest.json` exists but is incomplete (e.g.
  no runtimeAdapters or empty), treat as "first time" and re-run
  the install sweep. (This handles half-failed prior installs.)
- L1.6. Auto-install applies uniformly to:
  - `ael` (lobby entry)
  - `ael chat` (explicit lobby)
  - `ael <role>` shortcuts (pi / advisor / writer / ...) — these
    should also trigger auto-install when the project isn't ready,
    rather than failing with substrate errors.
- L1.7. `ael install ...` invoked EXPLICITLY by the user does NOT
  short-circuit through the auto-install path. Explicit install
  retains its current arg semantics.

### L2. Two-question lobby (role + runtime)

After L1's setup completes, the lobby asks:

1. **Role** (existing question, unchanged):
   ```
   AEL — 你想跟谁聊？ / who do you want to talk to?
     pi / 项目经理         ...
     advisor / 顾问        ...
     ... (9 roles)
   > _
   ```

2. **Runtime** (new question, conditional):
   ```
   AEL — which runtime?
     1) codex
     2) claude-code
     3) opencode
   > _
   ```

Acceptance criteria:

- L2.1. Build the runtime list dynamically from L1's install
  results, NOT from a hardcoded set. If only codex installed
  successfully (claude / opencode CLI not found), the runtime list
  is `[codex]`.
- L2.2. **If runtime list has exactly 1 entry, skip Q2 entirely**
  (Owner's Q1 hybrid choice). Print a one-line note like:
  ```
  → codex (only installed runtime)
  ```
- L2.3. If runtime list has ≥2 entries, ask Q2. Accept any of:
  - Exact slug: `codex` / `claude-code` / `opencode`
  - Short alias: `claude` → `claude-code`
  - Number: `1` / `2` / `3` matching the printed order
  - Empty enter → first runtime in the list (no scoring; just
    "first matched")
  - `q` / `quit` / `exit` / Ctrl-D → leave without launching
- L2.4. After both questions answered, `exec` into `ael talk
  --runtime <runtime> <role>` (reusing existing talk_cmd path —
  do NOT duplicate the talk logic).
- L2.5. The existing free-form keyword routing for Q1 ("我想反思" →
  advisor) stays intact. No regression.

### L3. Tests, docs, version

Acceptance criteria:

- L3.1. `tests/ael_wrapper.test.sh` cases:
  - `ael` in a fresh tmpdir with no runtime CLIs → exits non-zero
    with "no runtime found" message
  - `ael` in a fresh tmpdir with fake `codex` only → auto-installs
    codex, skips Q2, would `exec talk_cmd --runtime codex pi` (via
    test stub that captures the exec invocation)
  - `ael` in a fresh tmpdir with fake `codex` + `claude` → auto-
    installs both, asks Q2, accepts `claude` → routes to
    claude-code
- L3.2. `core/templates/ael-cli-reference.md` updated with the new
  lobby flow (PI / Advisor read this when Owner asks about CLI).
- L3.3. `install.sh` "Next:" hint changed from:
  ```

  ael                          # opens the lobby — pick who to talk to ...
  ```

  ```
  ael                          # auto-sets-up the team on first run, then opens the lobby
  ```
  (i.e. drop the `ael install` step — it's now auto.)
- L3.4. README.md + README.zh-CN.md Quickstart section updated to

- L3.5. CHANGELOG entry for v0.2.5 covering both bundles.

## Halt conditions (CEO stops + asks Owner)

- S1. If `ael install <runtime>` for a particular runtime takes
  >5 minutes (way too long), halt and ask whether to add a
  per-runtime timeout or accept the wait.
- S2. If detecting "is the manifest incomplete" gets tricky
  (different aiplus versions produce different manifest schemas),
  halt and ask for the canonical detection rule.
- S3. If runtime CLIs report different version-detection methods
  (`codex --version` vs `claude --version` vs `opencode --version`),
  halt and ask if uniform detection is needed (L1 may not require
  this — `command -v` is enough for "is it installed?").
- S4. If the existing `talk_cmd` doesn't cleanly accept the runtime
  arg from the lobby's exec path (e.g. there's an arg-ordering
  issue), halt — that's a Bug 2 regression risk and Owner needs to
  see it before we work around it.

## Anti-patterns

- Do NOT touch vendor/aiplus or any substrate-side logic. This is
  a wrapper-only sprint.
- Do NOT modify persona files in `core/templates/personas/*.md`.
  Lobby UX is wrapper concern; agent behavior is persona concern.
- Do NOT add Windows-specific changes to `bin/ael.ps1` in this
  sprint. Flag for v0.2.6 if the equivalent flow is needed there.
- Do NOT add interactive prompts INSIDE talk_cmd or other
  non-lobby code paths. The two-question UX lives in
  `chat_default` only.
- Do NOT auto-install during `ael install <args>`, `ael update`,
  `ael uninstall`, `ael status`, `ael doctor`, or `ael --version` —
  these are explicit user actions with their own contracts.
- Do NOT pre-emptively install runtimes the user doesn't have a
  CLI for. Owner's Q2 choice: silent skip.
- Do NOT change behavior for projects with an existing
  `.aiplus/manifest.json`. The new flow only activates on fresh
  projects.
