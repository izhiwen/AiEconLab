Companion to the short `/goal` text. CEO reads this fully before
starting.

## Mission

Owner dogfooded v0.2.3 on a real project (Qing_Ideology) and hit
three bugs within the first five minutes of trying to enter Advisor
via Claude Code. Each bug already has a workaround Owner can use, so
this sprint is **quality-of-life polish**, not blocking.

## Reproduction (verbatim Owner session)

```bash
$ ael install all
AIPLUS_UNEXPECTED_ERROR reason=uncaught detail=unknown --runtime 'all'.
AEL installed for runtime=all (chat tools registered)    # ← Bug 1: wrapper says success but substrate errored

$ ael advisor --runtime claude-code
Opening runtime=codex session as role=advisor ...        # ← Bug 2: --runtime flag was dropped, fell back to default codex

# (Then codex spent multiple turns trying to run a nonexistent
#  `aiplus identity --role advisor --runtime codex --with-memory ...`
#  before giving up and using a stale dev build.)        # ← Bug 3
```

## Scope (3 bundles)

### B1. `ael install all` must actually install all three runtimes

Acceptance criteria:

- B1.1. `ael install all` iterates over [codex, claude-code, opencode]
  and runs the full per-runtime install flow (install + add + set-team
  + mcp-register) for each. The wrapper must NOT pass the literal
  string `all` to the substrate — aiplus v0.6.8 rejects it.
- B1.2. If any runtime's install fails (e.g. claude-code CLI not on
  PATH), the wrapper prints a clear per-runtime status and continues
  to the next. Final summary lines show which runtimes succeeded.
- B1.3. Final summary is NOT a misleading "installed for runtime=all
  (chat tools registered)" when some failed. Use:
  ```
  AEL installed: codex ✓, claude-code ✓, opencode ✗ (CLI not on PATH)
  ```
- B1.4. Exit code: 0 only if all attempted runtimes succeeded; non-
  zero if any failed.
- B1.5. Test: add a `tests/ael_wrapper.test.sh` case asserting `ael
  install all --dry-run` mentions all three runtimes in the planned
  steps.

### B2. Role shortcuts must forward `--runtime` and other flags

Today the main()-case shortcut path:

```bash
pi|advisor|writer|ra-stata|ra-python|theorist|referee|replicator|pm)
  talk_cmd "$cmd"
  ;;
```

drops everything after the role name. Owner typing `ael advisor
--runtime claude-code` silently runs with whatever runtime got
auto-detected.

Acceptance criteria:

- B2.1. The shortcut path forwards remaining args to `talk_cmd`.
  Concretely: `ael advisor --runtime claude-code` must end up in
  `talk_cmd --runtime claude-code advisor`.
- B2.2. `talk_cmd`'s arg parser already handles `--runtime <X>` /
  `--runtime=<X>` before the positional role — just make sure the
  shortcut hands args in the right order.
- B2.3. Also support: `ael advisor "What is your role?"` —
  positional prompt after the role becomes a headless one-shot
  question (matches the existing `ael talk advisor "..."` semantics).
- B2.4. Tests in `tests/ael_wrapper.test.sh`:
  - `ael pi --runtime claude-code` is functionally equivalent to
    `ael talk --runtime claude-code pi` (verified via fake substrate
    that captures the runtime arg).
  - `ael advisor "hello"` is functionally equivalent to
    `ael talk advisor "hello"`.

### B3. **REVISED ROOT CAUSE** — `ael doctor` detects + offers to fix the aiplus version mismatch between `.aiplus/AGENTS.aiplus.md` and the `aiplus` binary on the user's PATH

**Original hypothesis was wrong.** The stale `aiplus identity --role
... --runtime ... --with-memory ...` call does NOT come from
`~/.codex/hooks.json` (those hooks point at `vibe-island-bridge` and
are fine). It comes from `.aiplus/AGENTS.aiplus.md` — a 706-line
agent-instruction file installed by `aiplus add aieconlab` /
`aiplus install <runtime>`. Line ~485 of that file literally tells
codex / claude / opencode to call `aiplus identity --role ... --runtime
... --with-memory ...` on session start.

**The bug is a version mismatch**, not a stale hook:

1. AGENTS.aiplus.md was generated by an aiplus version that supports
   the `--runtime` / `--with-memory` / `--emit-role-activated` flags
   on `aiplus identity` (verified working at v0.6.8 in advisor's
   manual test).
2. But many users have an older `aiplus` binary on their PATH (e.g.
   v0.5.22) that does NOT accept those flags. When the runtime obeys
   AGENTS.aiplus.md and runs the identity command, the OS-resolved
   `aiplus` rejects with `error: unexpected argument '--runtime' found`.
3. The runtime (codex specifically) then spends 10-30 minutes trying
   to recover — runs `aiplus identity --help`, finds dev builds in
   `~/Projects/AiPlus/...`, tries different flag combinations — before
   eventually giving up and engaging as the persona.

Owner reproduced this twice in 2026-05-19 dogfood:
- `ael advisor` → 0 minute hang (worked through it fast)
- `ael pi` → **27 minute hang** before PI engaged

Owner's environment had **three** `aiplus` binaries at three versions
(v0.5.22 system, v0.6.8 ael-support, v0.6.9-dev). The PATH-resolved
one was the oldest.

**Workaround (already applied by advisor in 2026-05-19)**: copy
`/Users/<u>/.local/libexec/ael-support` to
`/Users/<u>/.local/bin/aiplus` so PATH resolves to the bundled
v0.6.8. Verified: 27-minute hang disappears, `aiplus identity --role
pi --runtime codex --with-memory ...` returns IDENTITY_CONTEXT
immediately.

Acceptance criteria:

- B3.1. New check in `ael doctor`:
  - Run `command -v aiplus` to find the PATH-resolved binary.
  - Run `aiplus --version` to capture its version.
  - Run `ael-support --version` (or equivalent via substrate_bin()).
  - If they differ → print `NEEDS_FIX aiplus_version_mismatch` with
    both versions.
  - Additionally try `aiplus identity --help 2>&1 | grep -q
    -- --with-memory` — if missing the flag, print
    `NEEDS_FIX aiplus_identity_flag_missing`.
- B3.2. `ael doctor --fix` for this check is allowed:
  - Use `install -m 755 <ael-support-path> <aiplus-PATH-path>` to
    overwrite the PATH binary with the bundled substrate.
  - **Do NOT use `cp`** — on macOS it inherits `com.apple.provenance`
    xattr and SIGKILLs the binary on launch. Use `install` or
    `cat src > dst` (the latter only if RTK proxy isn't in the way).
  - Verify post-fix: rerun version check; rerun `aiplus identity
    --help` and confirm `--with-memory` is now in the output.
- B3.3. The fix is bounded: it only rewrites the PATH-resolved
  aiplus binary, never `~/Projects/AiPlus/*` or any path containing
  `dev` / `target/`. Print a clear "would write to X — proceed?"
  unless `--yes`.
- B3.4. Smoke check on POSIX only (macOS / Linux). On Windows we
  use install.ps1 path and the issue doesn't reproduce.
- B3.5. Test fixture: create a fake old `aiplus` binary in a tmpdir,
  put it on PATH ahead of ael-support, run `ael doctor`, assert the
  NEEDS_FIX line fires. Then `ael doctor --fix`, assert the fix
  replaces it, and verify `aiplus --version` matches ael-support.
- B3.6. Also do a one-line warning if AGENTS.aiplus.md is missing
  entirely (means `aiplus add aieconlab` didn't fully run) — but
  do not auto-regenerate (that's substrate's job).

## Halt conditions (CEO stops + asks Owner)

- S1. If B1's iterate-over-runtimes approach breaks an existing
  user's working setup (e.g. they were relying on `install all`
  failing fast on the first runtime), halt and ask whether to keep
  the new iteration behavior or add a `--strict` flag.
- S2. If B3's auto-fix (`install -m 755 <ael-support> <aiplus>`)
  fails for non-obvious reasons (codesign verification on macOS,
  read-only fs, etc.), halt and ask for guidance. The fallback is
  to print the manual command for Owner to run.
- S3. If a Mac fixture turns out to be required for Bug 1 testing,
  halt — we don't have a macOS CI runner.
