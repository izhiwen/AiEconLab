# AEL v0.2.6 · Auto-bypass everywhere — CEO autonomous goal

Companion to the short `/goal` text. CEO reads this fully before
starting.

## Mission

AEL users prioritize **efficiency over safety** (Owner-stated value,
2026-05-19). Every runtime ael launches should auto-enter bypass mode
so the user is not interrupted by approval prompts mid-flow.

Today's state (2026-05-19, after PR #123):

| Path | codex | claude | opencode |
|---|---|---|---|
| **Headless** (`ael advisor "Q"`) | ✅ bypass | ✅ bypass | ✅ bypass |
| **Interactive** (`ael` → lobby → role → runtime) | ❌ | ❌ | ❌ |

The interactive path goes through `aiplus agent talk --runtime X
role`, and aiplus does NOT expose a flag-passthrough surface — it
exec-replaces to the runtime CLI without any bypass arg. This sprint
closes that gap on both sides.

## Architectural choice

Two reasonable insertion points. **B (substrate change + ael flag)
is preferred** — keeps aiplus the single owner of "how runtimes are
launched" and avoids ael duplicating runtime-specific knowledge.

### Option A — ael bypasses aiplus for interactive talk (rejected)

ael's `talk_cmd` would stop calling `aiplus agent talk` and instead:
1. Capture identity context via `aiplus identity --emit-role-activated`
2. Directly `exec codex --dangerously-bypass-approvals-and-sandbox ...`

Rejected because: it duplicates aiplus's runtime-launch logic (model
selection, env injection, persona file resolution), drifts when
aiplus evolves, and breaks the abstraction line we just established.

### Option B — aiplus accepts `--bypass`, ael always passes it (chosen)

aiplus side: `aiplus agent talk --bypass` → substrate appends the
runtime-specific bypass flag when exec'ing the runtime CLI:

| Runtime | Flag appended |
|---|---|
| codex | `--dangerously-bypass-approvals-and-sandbox` |
| claude-code | `--dangerously-skip-permissions` |
| opencode | `--dangerously-skip-permissions` |

ael side: `talk_cmd` adds `--bypass` to the aiplus call by default;
env `AEL_BYPASS=0` opts out (rare safety case).

## Scope (2 bundles, 2 PRs in 2 repos)

### B1. aiplus substrate — `agent talk --bypass` flag

**This is in `izhiwen/AiPlus` repo, not AEL.** Owner or aiplus advisor
runs this part. CEO of AEL does NOT touch vendor/aiplus.

Acceptance criteria:

- B1.1. New flag `--bypass` on `aiplus agent talk`. Default: false.
- B1.2. When `--bypass` is set, aiplus appends the runtime-specific
  flag from the table above to the runtime CLI exec argv.
- B1.3. Also accept env `AIPLUS_BYPASS=1` as equivalent to `--bypass`
  (useful for wrappers that can't easily mutate argv).
- B1.4. Help text on `aiplus agent talk --help` shows the new flag
  with a one-line note: "Pass through the runtime CLI's bypass
  flag (codex: --dangerously-bypass-approvals-and-sandbox; claude
  & opencode: --dangerously-skip-permissions). Use with caution."
- B1.5. Test: per-runtime fake CLI fixture that asserts the bypass
  flag is in the exec argv when `--bypass` is set, and absent when
  not set.
- B1.6. Bump aiplus patch version. Tag + release.
- B1.7. After release, bump `vendor/aiplus` submodule in AEL to the
  new aiplus version (Owner or advisor handles vendor bump PR
  separately).

### B2. ael wrapper — pass `--bypass` by default

Acceptance criteria:

- B2.1. `talk_cmd` (interactive path, ael:1012) calls
  `run_substrate_interactive agent talk --bypass --runtime "$runtime"
  "$role"` when `AEL_BYPASS` is unset or set to anything other than
  `0` / `false` / `no`.
- B2.2. If `AEL_BYPASS=0` (or `false`/`no`), `--bypass` is NOT passed
  — gives Owner an opt-out for unusual sandbox cases.
- B2.3. Document `AEL_BYPASS` in `--help` output under "Environment
  variables" section. One-line:
  `AEL_BYPASS=0    disable runtime bypass (default: enabled)`
- B2.4. Document in `core/templates/ael-cli-reference.md` under a
  new "Safety / bypass" section.
- B2.5. Test: `tests/ael_wrapper.test.sh` adds a case that fakes
  ael-support and asserts the `--bypass` arg is in the captured
  argv when `talk_cmd` runs (and absent when `AEL_BYPASS=0`).
- B2.6. Do NOT remove the headless bypass flags already in place
  (PR #123 fixed codex; claude/opencode were already correct).
- B2.7. If vendor/aiplus is not yet bumped to the version that
  understands `--bypass`, the test must skip the assertion with a
  clear `# skipped: vendor aiplus pre-bypass` message rather than
  fail. CEO of AEL waits for the vendor bump before merging B2.

### B3. Windows native wrapper parity

`bin/ael.ps1` also has a talk path. Same treatment:

- B3.1. Default-pass `-bypass` (or equivalent argv) to the
  ael-support invocation.
- B3.2. Respect `$env:AEL_BYPASS = "0"` opt-out.
- B3.3. Headless paths in ps1 — verify they have the right flag for
  each runtime (codex / claude / opencode). Match the bash wrapper
  state after PR #123.

## Halt conditions

- S1. If aiplus side is blocked (advisor not available to ship the
  substrate change), halt and ask Owner whether to: (a) wait, (b)
  ship Option A (ael bypasses aiplus) as a temporary measure, or
  (c) defer to v0.2.7.
- S2. If a runtime's bypass flag changes between aiplus runtime
  versions (e.g. codex deprecates `--dangerously-bypass-approvals-
  and-sandbox`), halt and ask — do not silently fall back.
- S3. If `AEL_BYPASS=0` opt-out breaks an existing test (because
  interactive without bypass now hangs in test fixtures), halt and
  ask whether to add `--print` or `--print-once` to fixtures.

## Anti-patterns

- Do NOT bypass aiplus for the interactive path in ael (Option A
  was rejected for a reason — re-read the architecture section).
- Do NOT remove `AEL_BYPASS` opt-out. Owner explicitly wants a
  reverse-escape hatch for rare safety-needed cases.
- Do NOT touch personas (`core/templates/personas/*.md`).
- Do NOT bump version banners in same PR (Owner cuts v0.2.6 after
  merges land).
- Do NOT add `--safe` as a positional flag without env equivalent —
  env is the canonical way to flip security defaults.
