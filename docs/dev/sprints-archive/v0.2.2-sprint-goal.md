## Mission

Close the **middle-tier drift** between AEL's user-facing wrapper and the
aiplus substrate that landed in v0.6.0+. AEL wrapper currently exposes
roughly 15% of aiplus's CLI surface; v0.2.1 closed the highest-leverage
gap (MCP auto-register via PR #70). v0.2.2 closes the next-most-valuable
slice: **mid-tier commands users would discover and want**, plus
**automated detection of future surface drift**, plus **branding leak
cleanup**.

This sprint is *not* about adding new features the substrate doesn't
already have. It is about making aiplus capabilities reachable through
the AEL brand without users hitting "ael: unknown command".

## Scope (3 bundles, A/B/C)

### A. CLI surface drift detection — extend the drift workflow

Today `.github/workflows/aiplus-drift-check.yml` only counts commits
behind vendor pin. It does **not** detect new aiplus subcommands the
AEL wrapper doesn't expose.

Acceptance criteria:

- A1. New step in `aiplus-drift-check.yml` that:
  - builds the current aiplus binary at `vendor/aiplus/HEAD` AND at
    `vendor/aiplus/origin/main` (or just runs each `--help`)
  - extracts top-level subcommands from each
  - diffs them
  - any subcommand present in origin/main but not vendor HEAD →
    surface in the drift issue body under a "New aiplus subcommands"
    section
- A2. The drift issue body now has two sections:
  - "Commits in aiplus not yet in AEL" (existing)
  - "New aiplus subcommands AEL wrapper does not expose" (new)
- A3. Test: open a draft branch that artificially bumps aiplus to a
  pin that has a "new" subcommand; verify the workflow detects it.

### B. Expose mid-tier aiplus commands in AEL wrapper

After audit (Owner conversation, 2026-05-18), these aiplus commands
are valuable for users but currently invisible:

| Command | Why expose | New `ael` form |
|---|---|---|
| `aiplus secret-broker` | Owner's canonical secret store; users likely also have keyring/BWS keys to manage | `ael secret-broker [subcommand]` (thin passthrough) |
| `aiplus velocity` | Sprint velocity metrics — useful for PI dispatch decisions | `ael velocity [subcommand]` |
| `aiplus update` / `aiplus self update` | Substrate self-upgrade; users currently have no way to update without rerunning installer | `ael update` (passthrough) |
| `aiplus refresh` | Persona/template drift repair within installed project | `ael refresh` |
| `aiplus rollback` | Undo a bad install | `ael rollback` |

Acceptance criteria:

- B1. Each of the 5 commands above has a case branch in `ael` wrapper's
  `main()` that calls `run_substrate_visible` (or `_quiet` if no output
  expected) with the appropriate subcommand.
- B2. `usage()` lists them under "Advanced (you rarely need these once
  you are in chat)" so the natural-language entry stays primary.
- B3. Tests in `tests/ael_wrapper.test.sh` verify each command name is
  routable (does not return "unknown command").
- B4. Substrate output sanitization still applies — no `AiPlus` /
  `aiplus` / `AIPLUS` in any user-visible output from these commands.

### C. README + FAQ substrate leak cleanup

Multiple places in the AEL repo still mention `aiplus` literal in
user-facing surfaces. Sweep them all to `ael`.

Acceptance criteria:

- C1. `grep -rn 'aiplus' README.md README.zh-CN.md` returns ONLY
  the legitimate substrate references in code blocks /
  technical-architecture sections (clearly fenced as substrate-internal,
  not as user-facing commands).
- C2. No README FAQ entry tells the user to type `aiplus <something>`.
  Every user-instruction is `ael <something>`.
- C3. The `docs/` tree is OK to leave as-is — those are dev docs, not
  user-facing.
- C4. The `core/templates/personas/*.md` files: PI persona section 3.1
  currently emits `aiplus agent route ...` commands in its example
  output. Update those examples to `ael route ...` so PI in chat
  emits AEL-flavored commands users can paste.

## Halt conditions (CEO stops and asks Owner)

- S1. Any of the 5 mid-tier commands in B turn out to have non-trivial
  argument semantics that can't be passthrough'd (e.g., they take
  positional + flag args that conflict with ael wrapper's existing
  routing). Stop and ask: passthrough as-is via `ael substrate`, or
  defer to next sprint?
- S2. PI persona update for C4 breaks the persona-behavior tests in
  CI. Stop and ask: should the example commands stay aiplus-flavored
  with a comment explaining why?
- S3. A surfaced "new aiplus subcommand" turns out to be a private /
  internal command (e.g., `aiplus mcp-serve` is infra, not user). Stop
  and ask Owner to extend an "ignore list" in the drift workflow.

## Anti-patterns

- Do not add to bundle B beyond the 5 commands listed. Owner's audit
  picked these as "valuable to expose"; CEO does not have authority
  to expand the list mid-sprint.
- Do not rename `.aiplus/` directory to `.ael/`. That requires aiplus
  substrate cooperation and is a v0.3 conversation, not v0.2.2.
- Do not bump vendor pin during this sprint. Vendor bumps are their
  own decision triggered by the drift workflow.
- Do not modify `tests/persona-behavior.test.sh` to make C4 pass; if
  the persona tests fail, that's evidence of a problem to escalate.
- Do not cut a release at the end. Owner decides release cadence; CEO
  lands changes to main and reports.
