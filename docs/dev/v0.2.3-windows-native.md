# AEL v0.2.3 · Windows native support — CEO autonomous goal

Companion to the short `/goal` text. CEO reads this fully before
starting.

## Mission

v0.2.0 shipped a Windows preview that requires the user to manually
download `ael-v0.2.0-windows-x86_64.tar.gz`, extract it, and run the
**bash** wrapper from git-bash or WSL. That is not a "Windows release"
in any honest sense — it's a Linux/Mac binary blob with a Unix wrapper.

v0.2.3 closes the gap. After this sprint, a Windows user runs **one
PowerShell command** and gets a native `ael` command on PATH, with no
WSL or git-bash dependency:

```powershell
irm https://raw.githubusercontent.com/izhiwen/AiEconLab/main/install.ps1 | iex
ael
```

The user experience MUST match the macOS curl one-liner in every
substantive way:

- Same `ael` command name (resolves via .cmd shim → .ps1 implementation)
- Same lobby UX (`ael` no args → role picker)
- Same role shortcuts (`ael pi` / `ael advisor` / ...)
- Same commands (`ael install` / `ael update` / `ael uninstall` /
  `ael status` / `ael doctor` / etc.)
- Same auto-resolve-latest behavior (no hardcoded version)
- Same opt-in PATH-edit (PowerShell user profile or HKCU\Environment)
- Same first-run onboarding
- Same `ael-cli-reference.md` consultation behavior for personas
- Same MCP auto-register

Owner directive on safety-vs-efficiency: AEL targets researchers who
prioritize efficiency. PI auto-invokes via tool-use (already
implemented in v0.1.9). Do NOT add Windows-specific confirmation
prompts that the Mac side doesn't have.

## Scope (3 bundles)

### W1. `install.ps1` — PowerShell installer

Port of `install.sh` to PowerShell. Same behaviors:

Acceptance criteria:

- W1.1. `install.ps1` exists at repo root alongside `install.sh`.
- W1.2. `irm <url>/install.ps1 | iex` works as a one-liner.
- W1.3. Resolves latest release from `/releases/latest` redirect (mirrors
  install.sh PR #91 logic). Fallback to a `MINIMUM_SUPPORTED` version
  constant if the lookup fails, with a clear warning.
- W1.4. Downloads `ael-vX.Y.Z-windows-x86_64.tar.gz` from the GitHub
  release. Verifies sha256 against the .sha256 sidecar.
- W1.5. Extracts to `$env:LOCALAPPDATA\Programs\AEL\bin\` (wrapper) and
  `$env:LOCALAPPDATA\Programs\AEL\libexec\` (substrate binary).
- W1.6. Detects PATH. If install dir is not on PATH, **prints a one-line
  PowerShell command the user can copy** (same UX as install.sh manual
  path). Does NOT auto-edit PATH unless `-AddToPath` switch is passed.
- W1.7. `-AddToPath` switch: edits `HKCU\Environment` PATH to append
  the install dir + broadcasts `WM_SETTINGCHANGE` so new shells pick
  it up. Idempotent (don't double-append).
- W1.8. `-DryRun` switch: prints what would happen without writing.
- W1.9. No substrate brand leak (no `aiplus` / `AiPlus` / `AIPLUS` in
  user-visible output). Reuse the same `sanitize_substrate_output`
  regex in PowerShell.
- W1.10. Post-install hint matches install.sh: directs user to
  `cd MyProject && ael install` then `ael`.

### W2. `ael.cmd` + `ael.ps1` — native Windows wrapper

`ael.cmd` is a tiny shim (~5 lines) that delegates to `ael.ps1` so
users can type just `ael` from cmd.exe AND PowerShell.

`ael.ps1` is the port of the 1019-line bash `ael` wrapper. Same
functions and commands. Same lobby. Same PI persona dispatch.

Acceptance criteria:

- W2.1. `ael.cmd` exists at `bin/ael.cmd` in the Windows tarball.
- W2.2. `ael.ps1` exists at `bin/ael.ps1`. Implements all top-level
  subcommands: `install`, `update`, `uninstall`, `chat`, `talk`,
  `route`, `status`, `doctor`, `telemetry`, `invite`, `dismiss`,
  `integrate`, role shortcuts (`pi`, `advisor`, `writer`, etc., 9
  shortcuts).
- W2.3. Lobby (`ael` no args) behaves identically: prints the menu in
  English + Chinese, reads one line, routes by exact slug + keyword
  + free-form intent.
- W2.4. PI inside chat still calls `agent_route` / `agent_status` MCP
  tools (the substrate binary handles MCP; the wrapper just spawns
  codex / claude / opencode with the persona prompt).
- W2.5. Substrate sanitization: PowerShell regex replaces
  `\bAiPlus\b` → `AEL`, `\baiplus\b` → `ael`, `\bAIPLUS\b` → `AEL`
  in all substrate stdout/stderr piped to the user.
- W2.6. First-run onboarding (`.aiplus\agents\.ael-greeted` marker)
  works the same on Windows path conventions.
- W2.7. PI / Advisor cli-reference lookup uses
  `.aiplus\modules\aieconlab\core\templates\ael-cli-reference.md`
  (Windows path separator; same content).

## Halt conditions (CEO stops + asks Owner)

- S1. Pester or PowerShell-7-vs-PS-5 compatibility: if a Pester test
  needs Pester 5 but the runner ships Pester 3, stop and ask whether
  to vendor a newer Pester or pin to runner's default.
- S2. HKCU\Environment PATH edit doesn't take effect across all
  shells (cmd / PowerShell / Windows Terminal). Stop and ask whether
  to also write to system PATH or accept the limitation.
- S3. Codex / Claude Code / OpenCode runtime detection on Windows.
  These CLIs may behave differently on Windows (different binary
  names, different PATH lookup). Stop and ask if any runtime-specific
  hack is needed in `ael.ps1`.
- S4. The substrate binary (`ael-support.exe`) needs Visual C++
  redistributable or similar. Stop and ask if we ship the redist
  alongside or document the requirement.

## Anti-patterns

- Do NOT make Windows users install WSL. The point is native.
- Do NOT make Windows users install git-bash. PowerShell 5+ should
  be sufficient (it ships with Windows 10/11 by default).
- Do NOT bundle a second copy of every persona file in a
  Windows-specific path. The same `.aiplus\modules\aieconlab\` layout
  works on Windows.
- Do NOT add a `--confirm` flag to PI dispatch on Windows that the
  Mac side doesn't have. Owner's directive: efficiency over
  upfront confirmation.
- Do NOT bump vendor pin during this sprint. Vendor lives at the
  v0.2.0 cut (aiplus 08fa6f0) for the entire sprint.
- Do NOT touch `ael` (bash wrapper). It is the Mac/Linux artifact;
  Windows is independent.
