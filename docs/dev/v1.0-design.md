## 2. Category 1: aiplus must grow these protocols

Spec each one. AiPlus advisor will estimate based on this section.

### 2.1 `AIPLUS_WRAPPER_BRAND` env var
**Behavior**: When set, aiplus replaces all user-facing "aiplus" strings (lobby, status, errors, help — NOT talk prompt; that's team brand) with the wrapper brand.
**Default**: unset → "aiplus" (current behavior)
**AEL usage**: `export AIPLUS_WRAPPER_BRAND=AEL` in wrapper entry
**Fallback if missing**: silent default to "aiplus"
**Test**: spike — `AIPLUS_WRAPPER_BRAND=AEL ael-support agent status` should show "AEL" not "aiplus"

### 2.2 `AIPLUS_DEFAULT_TEAM` env var
**Behavior**: When set, aiplus uses this as active team without explicit `--team`.
**Default**: unset → `agent-team` (current behavior)
**AEL usage**: `export AIPLUS_DEFAULT_TEAM=aieconlab` in wrapper entry
**Interaction with `aiplus agent set-team`**: explicit `set-team` wins (persisted state); env var fills in when state absent
**Test**: spike — `AIPLUS_DEFAULT_TEAM=aieconlab ael-support agent talk pi` should work without `--team aieconlab`

### 2.3 team.toml `default_bypass` field ✅ IN FLIGHT (D3)
**Behavior**: When team.toml declares `default_bypass = true`, aiplus auto-applies `--bypass` (or its runtime equivalent) to `agent talk`.
**Default**: field absent → false (current behavior)
**AEL usage**: aieconlab team.toml gets the field
**Migration**: existing team.toml without field → backward compat (false)
**Status**: AiPlus advisor implementing
**Test**: spike — `ael-support agent talk pi` with aieconlab active should auto-bypass without explicit flag

### 2.4 team.toml `role_aliases` table ✅ MANDATORY (D2 = a)
**Behavior**: team.toml declares `[role_aliases]` table mapping freeform input → canonical role. aiplus does fuzzy match before falling back to role-not-found.
**Default**: table absent → no fuzzy match (current behavior)
**AEL usage**: aieconlab team.toml gets econ aliases (中文 + English phrases). AEL deletes `detect_role_from_input` (~70 lines bash)
**Test**: spike — `ael "我想反思 RD 设计"` should route to advisor; `ael "派给 RA 跑回归"` should route to ra-stata

### 2.5 `aiplus agent talk --resume <role>` ✅ IN FLIGHT
**Status**: AiPlus advisor's proposal at `aiplus-public/docs/proposals/talk-resume.md`. v0.6.15 sprint.
**AEL action**: When shipped, add `--resume` to ael talk passthrough (5 lines).

## 6. Acceptance workflows — these 3 must e2e pass before v0.3.0 dogfood passes

⚠️ **Depends on Phase 1 protocol #4**: Workflow A uses NL routing ("我想反思 RD 设计"). D2 = (a), so this requires the `role_aliases` protocol shipped on aiplus side.

### Workflow A: Advisor → PI → RA-Stata → Referee chain
```
ael                                  # opens lobby
> 我想反思 RD 设计                   # NL routes to advisor
... [advisor conversation about identification] ...
[ctrl-d or :quit]

ael pi                               # direct shortcut
> 把 RD bandwidth sensitivity 派给 ra-stata
... [PI dispatches] ...

ael ra-stata                         # do the assigned work
... [Stata regression session] ...

ael referee                          # self-audit
... [referee reviews the regression] ...
```
**Pass criterion**: each handoff preserves project context; no leaked "AiPlus virtual team" strings; bypass active without prompt; brand consistently "AEL" in all output.

### Workflow B: Replicator clean-room
```
cd ~/research/<paper-to-replicate>
ael replicator                       # opens replicator persona
> 复现这个论文的 Table 3 main results。原始代码不要看，从论文文本出发
... [replicator discusses identification, plans clean-room approach] ...

ael ra-stata                         # do the regression run yourself
> 跑 Table 3 的 RD specification，bandwidth 自己选
... [Stata session produces output] ...

ael replicator                       # back to replicator for comparison
> 我们的结果和论文 Table 3 比较：差异在哪？
... [replicator critiques discrepancies] ...
```
**Pass criterion**: replicator persona stays consistent across two ael invocations; ra-stata switch is clean; no context bleed between roles; bypass active.

### Workflow C: Writer drafting intro with advisor consultation
```
ael writer                           # opens writer persona
> 我们论文的核心 contribution 是 X 用 Y 方法识别 Z。帮我起草 intro 第一段
... [writer drafts] ...

> 第二段：讲文献空白
... [writer continues] ...

ael advisor                          # consult on framing
> 这段 intro 强不强？identification story 清晰吗？
... [advisor critiques framing] ...

ael writer                           # back to writer with advisor's feedback
> 按 advisor 的建议改第一段
... [writer revises] ...
```
**Pass criterion**: writer ↔ advisor switching is fluent; writer persona stays consistent; advisor critique is substantive (not generic); no leaked branding; bypass active.

## 7. Rollback criteria — when to abort v1.0 and retreat to v0.2.x

> Three triggers for abort (each independently sufficient). If ANY fires during Phase 2 or 3, freeze v1.0 work and ship v0.2.x patches instead.

| Trigger | Threshold | Action |
|---|---|---|
| Regression count | > 5 distinct user-visible bugs caught during dogfood that can't be fixed in 1 week | Freeze v1.0; v0.2.x continues |
| Time over budget | Phase 1+2 combined > 6 weeks (was 3-5) | Stop, audit what went wrong, decide continue/abort |
| Workflow failure | Any of Workflows A/B/C cannot e2e pass after 2 attempts to fix | Halt; one workflow failing == design assumption wrong |

**On abort**: tag `archive/v1.0-attempt`, cherry-pick salvageable improvements to v0.2.x, write post-mortem at `docs/dev/v1.0-postmortem.md`.

**Not an abort trigger**: minor cosmetic issues; missing nice-to-haves; aiplus protocol delays (just delay Phase 2 correspondingly).
